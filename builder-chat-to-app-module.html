1 Why this matters

You can already build custom assistants. The next leap: these tools now write working software — a self-marking quiz, a random-name picker, a revision flashcard app — from a plain-English description. No computer science degree, and often no setup.

This module is the map. You'll learn the three rungs from "a thing in the chat" to "a link my whole department uses", and build something live in two minutes.

2 The idea

Three rungs of "building"

Each rung is more capable and more effort. Match the rung to the need — most classroom tools only need rung 1 or 2.

1
Zero setup

Artefact / canvas

The AI generates a single interactive thing that runs right there in the chat — a calculator, a quiz widget, a sorting game. Describe it, use it, refine it. Nothing to install.

2
A file on your machine

Local app

The AI gives you a self-contained file (often a single .html) you save and open in a browser. Yours to keep, tweak and reuse offline.

3
Hosted on the web

Deployed app

Put that file online (Netlify, Vercel, GitHub Pages) so anyone with the link can use it. Powerful — and where the data-safety stakes rise sharply.

The judgement call: don't deploy when an artefact will do. A revision game for your class needs rung 1; only reach for rung 3 when others genuinely need access via a link — and only after the safety checks in B4.

Copy & run in an artefact/canvasBuild me a self-contained interactive web page: a 10-question multiple-choice quiz on the water cycle for Year 5. Show one question at a time, give instant feedback, keep a running score, and a "restart" button. Make it colourful and child-friendly. Single HTML file, no external libraries.
A live, clickable quiz beside the chat — pupils could use it immediately. Then you iterate in plain English: "add a timer", "make the wrong answers shake", "add five harder questions". You're directing software without writing a line of it. That's rung 1, and it's enough for a huge amount of classroom use.
4 Your turn

Build one live tool

Five minutes. Generate a small interactive tool as an artefact/canvas, then refine it twice in plain English.

A starter — adapt the subjectBuild a single-file interactive web tool: a random name picker for classroom use. Let me paste a list of names, then pick one at random with a fun animation, and an option to "remove after picking". Clean, large text, no external libraries.

Where to generate runnable things

  • Artifacts are Claude's strength here — interactive HTML/JS tools render live and are easy to iterate on. The best starting point for rung 1.
  • Canvas can produce and preview code; ask for a single self-contained HTML file you can also save locally (rung 2).
  • Canvas generates and previews web apps; download the file to keep it.
  • Copilot can write the code; for a live interactive preview you'll typically use one of the others, or run the file locally. Stays the approved tool if any real data is ever involved.

5 🛡️ Data-safety checkpoint

The higher the rung, the higher the stakes

  • Keep tools client-side and data-free. A quiz or picker should hold no personal data and save nothing to a server. Build them so the content lives in the file, not a database of pupils.
  • Don't paste real pupil data into the build prompt — use generic examples; let teachers type their own (local) lists at point of use.
  • Deploying (rung 3) makes it public. Anyone with the link can see it, and you may be processing data the moment a user types something in. Nothing pupil-facing or data-handling goes live without the B4 checks and IT/DPO sign-off.

Rules of thumb: prototype at rung 1, and treat every step toward "hosted" as a step that needs a data conversation.
